Auth

Phone OTP, with a clean swap path

Login is a phone number and a six digit code. Supabase phone auth handles the whole flow, and the SMS provider behind it is invisible to the client.

During dev: test phone numbers, no SMS provider

Supabase has built-in support for test phone numbers with static OTPs. In the dashboard under Authentication, Phone provider settings, define a set of test numbers, each mapped to a fixed code. Use your numbers and mine, plus a few obvious fakes for demos. Example:

+15550001111 = 123456
+15550002222 = 123456
(add our real numbers here too)

With test numbers configured, signInWithOtp and verifyOtp work end to end with no SMS being sent and no Twilio account needed. The full login screens can be built and demoed immediately.

Do not build a bypass

Do not implement "accept any 10 digit number with any code" anywhere in code, even behind a flag. A flag like that eventually ships to production. Supabase test numbers give the same convenience, scoped explicitly, with zero risk.

At launch: plug in Twilio Verify

When we are ready for real users, we add Twilio Verify credentials in the same Supabase Phone provider settings and remove the test numbers. Nothing in the client changes, because the client only ever calls Supabase auth.

Hard rule

The client talks only to Supabase auth methods. No code path in the app may reference Twilio, know the SMS provider, or implement its own OTP logic. The request-code and verify-code contract stays identical from dev through production.

Sessions and accounts

On first successful verification, create the account row keyed to auth.uid() and route the user into onboarding (screens 2, 2a or 2b in the flow document). On later logins, route straight to the chat home. Sessions should be long-lived on mobile; a cancer patient should not be re-entering codes weekly.

Supabase setup

Two projects, versioned schema, RLS everywhere

Projects and environments

  1. Create two Supabase projects: sage-dev and sage-prod. All day-to-day work happens in dev. Prod stays empty until launch.
  2. Use the Supabase CLI with migrations from day one. Every schema change is a migration file in the repo (supabase migration new ...), applied to dev while developing and to prod at release. No manual table edits in the prod dashboard, ever. This keeps the two projects identical and the schema reviewable in git.
  3. Environment config: the app reads SUPABASE_URL and SUPABASE_ANON_KEY from env (Expo public env vars). Dev builds point at dev, release builds at prod. The service role key exists only in edge function secrets, never in the client, never in the repo.

Row level security

Enable RLS on every table before writing any client code against it. The baseline policy for every user-owned table: a row is visible and writable only when its account_id equals auth.uid(). This means patient data isolation is enforced by the database itself, not by application code, which is exactly the posture we want for health data.

alter table patient_profiles enable row level security;

create policy "own rows only"
on patient_profiles for all
using (account_id = auth.uid())
with check (account_id = auth.uid());
Hard rule

No table ships without RLS enabled and a policy written. If a table ever needs cross-user access later (it should not in v1), that is a deliberate policy change reviewed together, not a default.

Storage

One private bucket, records, for uploaded reports and photos. Paths are per-user: records/{account_id}/{file_id}. Storage policies mirror the table rule: users can read and write only under their own account folder. The client displays files through short-lived signed URLs requested at view time. Nothing in Storage is ever public.

Secrets

All third-party keys (AI providers, transcription, and later Twilio) live in edge function secrets via supabase secrets set. If a key would need to exist in the client for a feature to work, the feature is designed wrong; route it through an edge function instead.

The edge function boundary

The client never calls a third party

Every call that involves a secret, an AI model, or an external API goes through a Supabase Edge Function. The client talks to Supabase auth, the database, Storage, and our own edge functions. Nothing else.

This gives us three things: API keys never touch the browser or app, we can switch providers without shipping a client update, and every AI interaction passes through one place where the safety layer runs.

Suggested functions

FunctionDoes
chatThe main conversation loop. Receives the message and conversation id, runs the safety check first, then calls the AI gateway with the system prompt, voice-layer context, and profile state. Streams the reply. Writes any profile updates the model extracts.
safety-checkRuns inside chat before the model responds. Classifies the inbound message against the clinician-authored red-flag list. On trigger, returns the escalation card payload instead of a chat reply.
extract-documentTakes an uploaded file id, pulls it from Storage, sends it to the AI gateway for extraction, returns plain-language findings for the patient to confirm. Writes to the profile only after confirmation.
summarize-visitTakes a finished visit transcript and produces the three sections: what the doctor said, what you do next, ask next time. Saves to the health log.
match-trialsBuilds a ClinicalTrials.gov v2 query from the profile (condition, biomarkers, extent, prior lines, ECOG, distance from stored lat and lng) and returns ranked candidates with plain-language framing.
transcribeProxies streaming audio to the transcription provider during visit recording.
Hard rule

If a feature seems to need a third-party key in the client, stop and route it through an edge function. No exceptions in v1.

The AI layer

One gateway, any provider

Do not couple the codebase to a single AI vendor. All model calls go through one internal module with one interface, and providers plug in behind it as adapters.

The gateway lives inside the edge functions as a small shared module. Every function that needs a model calls the same interface:

type AiTask = "chat" | "extract" | "summarize" | "classify";

interface AiRequest {
  task: AiTask;
  messages: Msg[];        // provider-neutral: {role, content}
  schema?: JsonSchema;    // when structured output is needed
  stream?: boolean;
}

async function complete(req: AiRequest): Promise<AiResult>

Behind complete(), an adapter per provider (Anthropic, OpenAI, Google to start) translates the neutral format into that vendor's API. Which provider and model handles which task is configuration, not code:

// config, editable without touching call sites
tasks:
  chat:      { provider: anthropic, model: claude-sonnet-4-6 }
  extract:   { provider: anthropic, model: claude-sonnet-4-6 }
  summarize: { provider: openai,    model: gpt-5-mini }
  classify:  { provider: google,    model: gemini-flash }

Rules that keep this genuinely swappable:

  1. Neutral message format everywhere. Call sites never construct provider-specific payloads. Only adapters know vendor APIs.
  2. Prompts are versioned files in the repo, not strings scattered through code. One folder, one file per task, with the voice-layer template variables (patient name, pronoun, relationship, perspective) interpolated at call time.
  3. Structured output through the schema field. Each adapter maps it to that vendor's mechanism (tool use, JSON mode, response schema). Call sites just receive parsed JSON.
  4. No provider-exclusive features in core paths. If a vendor-specific capability is genuinely worth it somewhere, isolate it in the adapter with a fallback, and flag it so we know where the coupling is.
  5. Log task, provider, model, latency, and token counts on every call, so we can compare quality and cost per task and reroute in config.
Data model

Accounts, profiles, and immediate saves

The account (keyed to auth.uid(), owned by the phone number) is separate from the patient profile, because the account holder may be a caregiver. Core tables:

TableKey fields
accountsid (auth.uid), holder_name, perspective (self or caregiver), relationship, created_at
patient_profilesaccount_id, patient_name, birth_year, gender, location (lat, lng, display), cancer_type, cancer_subtype, extent, metastasis_sites[], biomarkers[], prior_treatments[], progressed_on_treatment, prior_lines, ecog, comorbidities[], labs jsonb, care_team jsonb
conversations / messagesStandard chat persistence, account scoped
visitsaccount_id, transcript, summary jsonb (said, next, ask), recorded_at
health_logaccount_id, entry_type, payload jsonb, logged_at
uploadsaccount_id, storage_path, extraction jsonb, confirmed boolean

Three behaviors from the flow document, restated as build requirements: every profile field is nullable and every feature degrades gracefully when a field is missing; every interview answer writes to the database the moment it is given, so the interview is pausable with zero loss; and extraction results enter the profile only after the patient taps confirmation (confirmed = true).

Build order

Suggested milestones

  1. Shell and auth: Expo project, Supabase dev project, migrations, phone OTP with test numbers, onboarding screens 1 through 2b.
  2. Chat loop: chat edge function, AI gateway with one provider adapter, anchor question, personalized menu, message persistence.
  3. Safety layer: classify task wired in front of chat, escalation card, care team capture.
  4. Visit recorder: consent screen, streaming transcription, summarize-visit, health log.
  5. Interview and uploads: interview screens with immediate saves, extract-document with confirmation.
  6. Trials: match-trials function, results screens, plain-language explanations.

Content rules apply from the first commit: no em dashes anywhere in patient-facing text, plain language at a sixth grade reading level, and "I'm not sure" as an option on every interview question.

Appendix

Safety layer: the red-flag list, v0.9

The full rule set ships as sage-safety-rules-v0.9.json alongside this document. Load it as data in the classify function; do not hardcode symptoms in prompt strings.

Status: draft pending physician review

This list is drawn from published patient guidance (NCI, ASCO, and major cancer center "when to call your doctor" materials). It is a working v0.9 so the build is never blocked, but a physician must review and sign off before Sage is used by real patients. Track the review as a launch blocker.

How the tiers work

TierMeaningUI treatment
T1 emergencyLife-threatening now: breathing failure, chest pain, stroke signs, seizures, uncontrolled bleeding, severe allergic reactionEscalation card leading with Call 911. No chat reply before the card.
T2 urgentNeeds the care team within the hour: any fever on treatment, cord compression signs, clot signs, severe GI, bleeding, port infectionEscalation card leading with Call your care team right away, 911 second, then offer to log the symptom.
T3 same dayWorth a call today but not a siren: persistent nausea, mild new swelling, days of poor intakeSoft banner above the normal chat reply. Conversation continues.
MH crisisSelf-harm or suicidal contentCompassionate message with crisis resources (988 in the US), not the medical card. Never discuss methods. Keep the conversation open.

Classifier behavior

The classify task runs on every inbound message before the chat model, receiving the message plus three context flags: on active treatment, perspective, patient name. It returns a tier, category, confidence, and one-line rationale. NONE proceeds to normal chat. Two rules govern edge cases: when in doubt between tiers, pick the higher one, and being on active treatment raises borderline fever, GI, and bleeding cases from T3 to T2. The single most important rule in the file: any fever at or above 100.4 F (38 C) during or within six weeks of chemo is T2, because it can be neutropenic fever.

Use the AI gateway classify task for this (a fast model is fine), with the JSON rules injected as reference data and the output constrained to the schema in the file. Log every non-NONE classification with the message, tier, and rationale so we can audit false positives and misses weekly during early testing.

AI judgment beyond the list

The rules are the floor, not the ceiling. The classifier is instructed to treat the list as examples that anchor its judgment: if a message sounds medically dangerous and no listed rule applies, it still assigns the appropriate tier from general clinical knowledge, and the same escalation card fires. The guiding question in the prompt: would a nurse hearing this tell the person to seek care, and how fast? These calls return rule_matched: false, are logged to a dedicated table, and are reviewed weekly. Recurring patterns get promoted into the rules file as named entries, so the list grows from real usage. Combinations matter too: the classifier weighs symptoms together, since mild items in combination (dizziness plus dark stools, for example) can signal something urgent that neither would alone.

The emergency number is a config value, 911 by default, so launching outside the US later is a settings change, not a code change.
